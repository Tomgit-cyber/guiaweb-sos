🎯 Objetivo

Organizar e orientar o desenvolvimento do aplicativo SOS Unimed Vidas em todas as suas fases e tecnologias integradas, servindo como documentação centralizada para garantir consistência e qualidade no processo de construção.

Este guia também prepara a transição para o setor de TI da Unimed Cachoeiro do Itapemirim, contendo todas as informações necessárias para manutenção e evolução do projeto.

📌 Fases do Projeto

As fases devem ser seguidas sequencialmente, com revisões periódicas entre cada etapa. O status atual está destacado nas checkboxes.

🧰 Ferramentas Utilizadas

  • GitHub + Vercel 🔍Configuração de deploy automático documentada em /docs/deploy.md
  • FlutterFlow 🔍Projeto compartilhado com equipe de TI: SOS-Vidas-FF
  • Figma 🔍Design System completo: link no README
  • Formspree 🔍Configuração de formulários em /src/forms

🚀 Fluxo Padrão de Deploy

  1. Atualização de arquivos no GitHub (branch `main`)
  2. Revisão de código via Pull Request (mínimo 2 aprovações)
  3. Deploy automático no Vercel (gatilho por push)
  4. Verificação no link publicado (ambiente de staging)
  5. Testes de regressão (script `npm run test-e2e`)
  6. Deploy em produção (manual pelo Vercel)
  7. Comunicação à equipe (Slack #sos-deploys)

Qualquer alteração no fluxo deve ser documentada e comunicada a todos os envolvidos. Em caso de erro, seguir protocolo em /docs/emergency.md.

🧠 Boas Práticas

  • Use commits claros e descritivos (conventional commits)
  • Documente tudo no README e Wiki do projeto
  • Evite código sem função ou comentado
  • Utilize branches por tarefa/feature (prefixo: feat/, fix/, docs/)
  • Realize revisões de código periódicas
  • Mantenha padrões de acessibilidade (WCAG 2.1 AA)
  • Testes unitários para funções críticas (Jest)
  • Versionamento semântico (SemVer)

As boas práticas são auditadas mensalmente pelo script `audit-code-quality.js`.
